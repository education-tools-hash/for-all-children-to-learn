# どのまな 開発ルール Ver.1.0

最終更新: 2026-07-24  
適用範囲: どのまな公式サイト・公開中の各ブラウザアプリ・今後追加する新規ページ／新規アプリ  
管理対象: GitHub Pages 上で公開する HTML / CSS / JavaScript ファイル

---

## 0. この文書の目的

本書は、「どのまな」の開発や修正を、人間・Claude Code・その他のAI・将来の共同開発者が行う際に、判断基準を統一するためのルールです。

目的は、単にコードの書き方をそろえることではありません。

- 「どのまな」らしい見た目と使い心地を守る
- 既存機能を壊さず、安全に改善する
- 特別支援教育の現場で、安心して使える状態を保つ
- AIが必要以上に変更したり、勝手に技術構成を変えたりすることを防ぐ
- 開発に詳しくない運営者でも、変更内容を理解し判断できるようにする

判断に迷った場合は、次の優先順位で決めます。

1. 利用者の安全とアクセシビリティ
2. 既存機能を壊さないこと
3. 「どのまな」の理念と世界観
4. 操作の分かりやすさ
5. 既存コードとの一貫性
6. 技術的な新しさ

# 1. どのまなの理念

## 1.1 ブランドメッセージ

> **どの子も、学べる。**

「どのまな」は、障害の有無や年齢だけで利用者を限定せず、一人一人の学び方や表現の仕方を尊重する、無料のブラウザ教材・ICT活用支援プラットフォームです。

## 1.2 開発思想

「その子に合わせる」のではなく、

> **その子のやり方を尊重する。**

ことを大切にします。

利用者を、決められた一つの操作方法に合わせるのではありません。

- タッチ
- マウス
- キーボード
- 視線入力
- 外部スイッチ
- 音声
- 支援者との共同操作

など、多様な方法で利用できる可能性を残します。

## 1.3 どのまならしさ

すべてのページとアプリは、次の印象を目指します。

- やさしい
- 安心感がある
- 静かで落ち着いている
- 分かりやすい
- 誰でも使いやすい
- あたたかい
- 信頼できる
- 子どもっぽすぎない
- 教育現場で使いやすい
- 長く使える

## 1.4 避ける印象

- 派手すぎる
- 広告のように見える
- 情報量が多すぎる
- 操作を急かされる
- 幼児向けに見えすぎる
- 説明が専門的すぎる
- 過度に先進的・機械的に見える
- 見た目を優先して使いにくくなる

# 2. 現在の技術構成

## 2.1 採用技術

現在の「どのまな」は、原則として次の構成を維持します。

| 項目 | 採用内容 |
|---|---|
| 公開方式 | GitHub Pages |
| 基本言語 | HTML5 |
| スタイル | CSS3 |
| 動作 | Vanilla JavaScript |
| データ | JSON等の静的ファイル |
| ビルド処理 | 原則不要 |
| サーバー処理 | 原則使用しない |
| 対応端末 | PC / タブレット / スマートフォン |

## 2.2 技術構成の変更禁止

Claude Codeやその他のAIは、明示的な承認なしに以下を行ってはいけません。

- Reactへの移行
- Next.jsへの移行
- Vue、Svelte等への移行
- TypeScriptへの全面移行
- Tailwind CSSの導入
- npmやNode.js前提の構成への変更
- ビルド工程の追加
- 大規模なディレクトリ再編
- 既存ページのSPA化
- 外部CMSやデータベースの導入

これらは将来検討する可能性がありますが、通常の改善作業として勝手に行ってはいけません。

## 2.3 外部ライブラリ

- 新しい外部ライブラリは、原則追加しない
- 既存ライブラリで実現できる場合は、既存のものを使う
- CDNの追加は、セキュリティ・読み込み速度・長期運用への影響を確認する
- 依存関係を追加する場合は、実装前に運営者の承認を得る

# 3. 開発の基本原則

## 3.1 既存機能を壊さない

最優先事項です。

修正前に、対象ページの以下を確認します。

- リンク
- ボタン
- 音声
- 動画
- 保存機能
- 印刷機能
- ダウンロード機能
- 外部スイッチ操作
- キーボード操作
- スマートフォン表示
- タブレット表示
- PC表示

見た目を改善する修正で、機能を変更してはいけません。

## 3.2 変更範囲を限定する

- 指示されたファイルだけを変更する
- 関係のないコードを整理しない
- 「ついでの修正」を行わない
- 大規模なリファクタリングは別タスクに分ける
- 1回の作業は、原則として1つの目的に限定する

## 3.3 変更前の確認

Claude Codeは、作業前に以下を確認します。

1. 対象ファイル
2. 現在の構造
3. 関連するCSS・JavaScript
4. 既存の共通デザイン
5. 影響を受けるページ
6. 既存機能の確認方法

## 3.4 分かりやすい説明

開発に詳しくない人でも理解できるように説明します。

専門用語だけで説明してはいけません。

悪い例:

> DOM構造とCSS specificityを修正しました。

良い例:

> 文章の幅が画面ごとに変わっていたため、共通の最大幅を設定しました。既存のリンクやボタンの動作は変更していません。

# 4. ホームページとアプリの区別

「どのまな」では、公式サイトと学習アプリで目的が異なります。

## 4.1 公式サイト

公式サイトの目的は、利用者に安心してもらい、必要な教材へ迷わず移動してもらうことです。

優先順位:

1. どのまなの理念
2. どのような人が使えるか
3. 特徴
4. アプリカテゴリ
5. おすすめアプリ
6. 更新情報
7. お問い合わせ
8. 運営者・利用規約等

公式サイトでは、派手な演出よりも、信頼感・分かりやすさ・読みやすさを優先します。

## 4.2 学習アプリ

学習アプリの目的は、利用者が説明書なしでも活動を始められ、学習や余暇活動に取り組めることです。

原則:

- 3秒程度で何をする画面か理解できる
- 操作対象が分かりやすい
- 開始方法が分かりやすい
- 終了・戻る方法が分かりやすい
- 誤操作しても簡単に戻せる
- 音がなくても可能な限り利用できる
- 正解・不正解だけでなく、操作したこと自体が分かる
- 支援者が横から補助しやすい

## 4.3 アニメーションの扱い

### 公式サイト

- 装飾目的の派手な動きを使わない
- スクロールを妨げない
- 自動再生の長い演出を使わない
- 状態変化を伝えるための控えめな動きに限定する

### 学習アプリ

次の場合は、楽しさを目的とした動きも使用できます。

- 因果関係を理解する
- 注視を促す
- 選択したことを伝える
- 達成感を伝える
- 学習意欲を高める
- 感覚遊びや視線入力活動として必要

ただし、次は避けます。

- 激しい点滅
- 予測できない急な動き
- 長時間止められない動き
- 過度なバウンス
- 強い振動表現
- 利用者を急かすカウントダウン
- 音を消せない自動再生

# 5. デザインルール

## 5.1 基本原則

1. 迷わせない
2. 一貫性を優先する
3. 余白で情報を整理する
4. 見た目より使いやすさを優先する
5. 端末ごとに崩れない
6. 子ども向けにしすぎない
7. 文字と操作対象を十分に大きくする

## 5.2 レイアウト

- コンテンツ最大幅は、原則 `1120px` 前後
- 本文の最大幅は、読みやすさを考慮し `38em` 前後
- スマートフォンの左右余白は `16px` 以上
- PCの左右余白は `24px` 以上
- セクション間は十分な余白を設ける
- 罫線や囲みを増やす前に、余白で整理できないか検討する

## 5.3 スペーシング

余白は、原則として4pxまたは8pxの倍数を使用します。

| 用途 | 目安 |
|---|---|
| アイコンと文字 | 4〜8px |
| 要素内の小さな間隔 | 8〜12px |
| ボタン間 | 12〜16px |
| カード内余白 | 16〜24px |
| 要素間 | 24〜32px |
| セクション間（スマホ） | 48px |
| セクション間（PC） | 64〜80px |

## 5.4 フォント

優先フォント:

```css
font-family:
  "BIZ UDゴシック",
  "BIZ UDPGothic",
  "Noto Sans JP",
  "Yu Gothic",
  "Meiryo",
  sans-serif;
```

原則:

- 本文は16px以上
- 主要なアプリ操作では18px以上も検討する
- 本文の行間は1.6〜1.8
- 見出しの行間は1.3〜1.5
- 細すぎるフォントウェイトを使わない
- 長文で中央揃えを多用しない
- 見出し階層を飛ばさない

## 5.5 角丸

「どのまな」のやさしさを表現するため、適度な角丸を使用します。

| 用途 | 目安 |
|---|---|
| 小さな入力欄・小ボタン | 10〜12px |
| 標準ボタン | 12〜16px |
| カード | 16〜20px |
| 大きな案内カード | 20〜24px |
| バッジ | 999px |

丸くしすぎて幼児向けに見えないようにします。

## 5.6 影と境界線

- 強い影を使わない
- カードは薄い境界線またはごく薄い影を使う
- 境界線と影を両方強く使用しない
- 重要度を影の濃さだけで表現しない

推奨例:

```css
box-shadow: 0 2px 8px rgba(20, 50, 60, 0.08);
border: 1px solid var(--color-border);
```

# 6. 配色

## 6.1 原則

- 既存のロゴ・Design System・公式資料に定義された色を優先する
- Claude Codeが独自に新しいブランドカラーを決めない
- 色はCSS変数で管理する
- 同じ意味には同じ色を使う
- 色だけで意味を伝えない

## 6.2 推奨する役割

実際のカラーコードは、既存のDesign Systemの値を優先します。

```css
:root {
  --color-primary: /* どのまなのメインカラー */;
  --color-primary-dark: /* hover・強調 */;
  --color-primary-light: /* 薄い背景 */;

  --color-secondary: /* 補助色 */;
  --color-accent: /* オレンジ等のアクセント */;

  --color-text: #24343a;
  --color-text-sub: #5f6f74;
  --color-bg: #ffffff;
  --color-bg-soft: #f7faf9;
  --color-border: #dfe8e6;

  --color-success: #2f855a;
  --color-warning: #b7791f;
  --color-danger: #c53030;
  --color-info: #2b6cb0;
}
```

## 6.3 禁止事項

- コンポーネントごとに別の青や緑を追加する
- 既存のロゴ色を装飾として乱用する
- ロゴの4色をそのまま別々のアプリアイコンに流用する
- コントラストが不足する薄い文字色
- 赤と緑だけで正誤を表す

# 7. ボタンと操作要素

## 7.1 タップ領域

- 最低44×44px
- 学習アプリでは48×48px以上を推奨
- 隣り合うボタンは十分に離す
- アイコンだけのボタンには必ず説明を付ける

## 7.2 主要アクション

「1画面に主要ボタンは必ず1つ」というルールは、一覧画面には適用しません。

正しくは次の通りです。

- 1つの操作領域につき、主要アクションは原則1つ
- 1つのカードにつき、主要アクションは原則1つ
- アプリ一覧では、各カードに同じ種類の主要ボタンを置いてよい
- 破壊的な操作を通常の主要ボタンと同じ見た目にしない

## 7.3 ラベル

曖昧な文言を避けます。

悪い例:

- OK
- 決定
- 実行
- 送信

良い例:

- 学習をはじめる
- 保存する
- 印刷する
- ホームに戻る
- 音を止める
- もう一度取り組む

## 7.4 HTML要素

- 操作には `<button>` を使う
- ページ移動には `<a>` を使う
- `<div>` にクリック処理だけを付けてボタン化しない
- 無効状態では理由が分かるようにする
- キーボードフォーカスを消さない

# 8. アクセシビリティ

アクセシビリティは追加機能ではなく、品質の下限です。

## 8.1 必須項目

- 文字と背景のコントラストを確保する
- 本文は原則4.5:1以上
- 色だけで状態を伝えない
- キーボードだけで主要機能を操作できる
- フォーカス位置が視覚的に分かる
- 画像に適切な `alt` を付ける
- 装飾画像は `alt=""`
- フォーム要素にラベルを付ける
- 音声情報に代替手段を検討する
- 動画には字幕や説明を検討する
- 自動再生音を避ける
- 拡大表示で操作できる
- 横スクロールを原則発生させない

## 8.2 外部スイッチ・支援機器

- 標準的なHTML要素を使う
- フォーカス順を自然にする
- 操作対象を小さくしすぎない
- 長押しを必須にしない
- ドラッグ操作だけを必須にしない
- ホバーだけで情報を表示しない
- 一定時間内の操作を強制しない
- 誤操作しても戻れるようにする

## 8.3 動きへの配慮

`prefers-reduced-motion` に対応します。

```css
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
```

学習上必要な動きまで完全に消す場合は、代替の視覚フィードバックを用意します。

# 9. レスポンシブ対応

## 9.1 確認幅

最低限、次の幅で確認します。

| 幅 | 想定 |
|---|---|
| 375px | スマートフォン |
| 768px | タブレット縦 |
| 1024px | タブレット横・小型PC |
| 1280px | PC |

## 9.2 必須確認

- 文字が不自然な位置で分割されない
- 中黒「・」の前後で不自然に折り返さない
- ボタン文字が2列に崩れない
- 二次元コードが枠からはみ出さない
- カード同士が重ならない
- 画像が引き伸ばされない
- 横スクロールが発生しない
- 固定ヘッダーが本文を隠さない
- 画面回転後も操作できる

## 9.3 日本語の折り返し

必要に応じて、次を使用します。

```css
word-break: auto-phrase;
overflow-wrap: anywhere;
line-break: strict;
```

ただし、ブラウザ対応状況を確認し、語の途中で不自然に分割される場合はHTML構造や要素幅も見直します。

# 10. HTML・CSS・JavaScriptのルール

## 10.1 HTML

- セマンティックなHTMLを使う
- ページごとに原則1つの`h1`
- 見出し階層を守る
- ボタンとリンクを正しく使い分ける
- `lang="ja"`を設定する
- ビューポート設定を入れる
- 画像サイズを可能な限り指定する
- 重複する`id`を作らない
- インラインイベント属性を増やさない

## 10.2 CSS

- 色・角丸・余白などはCSS変数にまとめる
- 既存の共通スタイルを優先する
- `!important`を増やさない
- 固定幅を多用しない
- `position: absolute`だけでレイアウトを組まない
- メディアクエリを必要以上に乱立させない
- 同じスタイルを複数ファイルに重複させない
- 印刷ページでは`@media print`を確認する

## 10.3 JavaScript

- 既存機能を理解してから修正する
- グローバル変数を増やしすぎない
- 同じ処理を重複させない
- エラーを握りつぶさない
- 不要な`console.log`を残さない
- DOM取得後の存在確認を行う
- 多重クリックや二重実行を防ぐ
- ローカル保存を変更する場合は、既存データとの互換性を考慮する

## 10.4 コメント

コメントは「何をしているか」だけでなく、「なぜ必要か」を書きます。

悪い例:

```js
// ボタンを取得
```

良い例:

```js
// 外部スイッチからも操作できるよう、標準button要素のままイベントを追加する
```

- 古いGitHub Pages URLを新規ページに記載しない
- URL変更時はサイトマップや内部リンクへの影響を確認する

構造化データを追加する場合は、実際のページ内容と一致させます。

# 12. パフォーマンス

- 画像は適切なサイズに圧縮する
- 写真はWebP等も検討する
- ロゴ・アイコンは可能な限りSVGを使う
- 二次元コードは拡大しても劣化しない形式を優先する
- 画面外画像は`loading="lazy"`を検討する
- ファーストビュー画像は遅延読み込みしすぎない
- 不要なJavaScriptを読み込まない
- 外部フォントの読み込み過多を避ける
- レイアウトシフトを防ぐ

Lighthouseの点数だけを目的に、必要な機能やアクセシビリティを削ってはいけません。

# 13. Git・GitHub運用

## 13.1 基本

- `main`ブランチは公開用
- 作業前に最新状態を確認する
- 変更前の状態に戻せるようにする
- 1回のコミットは1つの目的にする
- デザイン変更と機能変更を可能な限り分ける
- 大量のファイルを一度に変更しない

## 13.2 Claude Codeに禁止するGit操作

明示的な承認なしに行ってはいけません。

- `git push`
- `git push --force`
- ブランチ削除
- 履歴の書き換え
- `reset --hard`
- 大量ファイルの削除
- mainへの直接マージ
- タグ作成
- 公開設定の変更

## 13.3 コミットメッセージ

日本語で分かりやすく記載して構いません。

例:

```text
fix: トップページのスマホ表示崩れを修正
style: アプリカードの余白と角丸を統一
docs: どのまな開発ルールを追加
```

# 14. Claude Codeへの指示ルール

## 14.1 Claude Codeの役割

Claude Codeは、設計者ではなく実装支援者です。

- 運営者の目的を勝手に変更しない
- ブランドの方向性を勝手に再定義しない
- 技術構成を勝手に変更しない
- 指示範囲外を勝手に改善しない
- 不要な質問を繰り返さない
- 危険な変更だけは必ず確認する

## 14.2 作業前に行うこと

最初に、実装せず次を日本語で報告します。

1. 現在の状態
2. 変更予定の内容
3. 変更するファイル
4. 変更しない箇所
5. 想定される影響
6. 確認方法

ただし、運営者が「そのまま実装してください」と明示した場合は、計画を簡潔に示して実装へ進んで構いません。

## 14.3 質問してよい条件

Claude Codeが質問するのは、原則として次の場合に限ります。

- 既存機能を削除する可能性がある
- 公開中のURLやページ構成が変わる
- Git操作や依存パッケージ追加が必要
- 複数案で結果が大きく異なる
- 指示範囲外のファイルを変更する必要がある
- 個人情報・認証情報・本番設定に関わる
- 元に戻しにくい変更が発生する

軽微な実装判断は、既存コードと本ルールに従って進め、作業後に報告します。

## 14.4 Yes / No確認を求める場合

Yes / Noだけを求めてはいけません。

必ず日本語で次を説明します。

```text
この操作では〇〇を変更します。
既存の△△には影響しません。
元に戻すことはできます。
実行してよければ「Yes」、中止する場合は「No」です。
```

危険な操作の場合:

```text
この操作は公開中のmainブランチを書き換えます。
誤ると公開ページに影響します。
現時点では実行せず、別ブランチで確認することを推奨します。
```

## 14.5 指示テンプレート

Claude Codeへの依頼は、次の形式を推奨します。

```text
【目的】
利用者にとって、どのように改善したいかを書く。

【対象】
変更してよいファイルを書く。

【変更しないもの】
既存機能、URL、文章、画像など、変更してはいけないものを書く。

【デザイン方針】
どのまな開発ルール Ver.1.0 と、既存のDesign Systemに従う。

【完了条件】
どの状態になれば完了かを書く。

【確認】
スマートフォン、タブレット、PCで表示を確認する。
既存リンクと主要機能が動作することを確認する。
```

## 14.6 Claude Codeへの禁止事項

1. 指示範囲外のファイル変更
2. 勝手なリファクタリング
3. 外部ライブラリ追加
4. React・Next.js等への移行
5. 新しいブランド色の追加
6. 既存ロゴの改変
7. 文章の意味を勝手に変える
8. リンク先やファイル名の変更
9. テストや確認をせず完了報告
10. git push等の公開操作
11. `.env`や認証情報へのアクセス
12. 動作中の機能を見た目のために削除
13. 「不要だと思った」という理由で要素を削除
14. 不明点を勝手な推測で大きく補う
15. 変更内容を英語だけで報告する

## 14.7 作業後の報告

作業後は必ず日本語で、次の形式で報告します。

```text
【変更した内容】
・

【変更したファイル】
・

【変更していないもの】
・

【確認したこと】
・スマートフォン表示
・タブレット表示
・PC表示
・リンク
・主要機能
・キーボード操作

【未確認・注意点】
・

【元に戻す方法】
・
```
